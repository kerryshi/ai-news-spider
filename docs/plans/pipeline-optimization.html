Phase 1 — Storage & query wins

recommendedlow risk · high value
  • Indexes on items(enriched), items(first_seen), items(last_seen), items(source) (guarded CREATE INDEX IF NOT EXISTS).
  • Kill the N+1: rank() calls engagement_series() once per item. Velocity only needs the first & last snapshot → one GROUP BY query for all ids.
  • Trim get_corpus: push the since window into SQL; skip parsing embeddings unless query-mode needs them (saves parsing ~2,889 × ~15KB JSON every top).
  • Batch commits: upsert_item commits per item → one transaction per collect phase.
Expected: warm top latency ↓ large (esp. embedding-skip), collect upsert time ↓, prune/get_corpus ↓.

Phase 2 — Enrichment throughput (raise the ceiling)

medium risk
  • Vectorize novelty with numpy: load prior embeddings once as an (N×768) array; cosine via one matmul instead of pure-Python loops over up to 4,000 vectors per new item. Needs numpy on the Jetson — see decisions.
  • Parallelize the LLM calls: embed + judge are network-bound; run them concurrently (thread pool) against the desktop Ollama. DB writes stay single-threaded (SQLite). Lifts the ~219/cycle ceiling.
  • Binary embeddings (optional): store float32 bytes instead of JSON TEXT — smaller, faster load. Data-model migration — see decisions.
Expected: enrich wall-time ↓ (numpy + concurrency), ceiling ↑ ~2–4× → more net-new per cycle.

Constraints & validation

  • No behavior change to ranking semantics or digest output; numpy/Ollama-absent paths still degrade gracefully.
  • Guarded migrations (CREATE INDEX IF NOT EXISTS, dual-read) so existing state.db upgrades in place.
  • Tested on the desktop .venv (suite is 60 green); every change adds a regression test; novelty numpy-vs-python parity test.
  • Deploy via deploy.ps1 (Jetson is a file copy); validate with a real collect cycle + the Phase 0 bench before/after.

Risks / unknowns

  • numpy availability/perf on the Jetson (old aarch64). Phase 0 verifies before Phase 2 commits to it.
  • Ollama concurrency vs. other GPU workloads on the desktop — start at 4, measure.
  • SQLite single-writer: parallelize only the network calls, keep DB writes serial.
